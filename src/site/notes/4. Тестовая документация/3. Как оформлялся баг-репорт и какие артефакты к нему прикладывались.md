# 3. Как оформлялся баг-репорт и какие артефакты к нему прикладывались
Я оформлял баг-репорты так, чтобы разработчику было легко воспроизвести и локализовать проблему.

В баге указывал краткое описание / заголовок, окружение, версию сборки, предусловия, шаги воспроизведения, ожидаемый результат, фактический результат, серьёзность, приоритет, тестовые данные и ссылки на связанную задачу или тест-кейс.

Из артефактов обычно прикладывал скриншоты, видео, HAR-файл, логи, ответ API, payload запроса, данные из БД, `request_id`, `trace_id`, `operation_id`. Если проблема была в UI — прикладывал скрин или видео. Если в API или интеграции — запрос, ответ, код ошибки, логи и ID операции.

Например, если платёж в UI отображается как успешный, а в БД остаётся `Failed`, я приложу скрин из UI, ответ API, запись из БД, логи backend-а и ID операции. Так разработчик сразу видит, что проблема не просто “на экране”, а в рассинхроне между слоями системы.

В финтехе особенно важно прикладывать технические артефакты, потому что многие дефекты связаны не с UI, а с backend-логикой, интеграциями, брокерами сообщений и статусами операций.
