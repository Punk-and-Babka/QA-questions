# 18. Как применять классы эквивалентности, граничные значения и попарное тестирование при проверке параметров и их комбинаций

Я бы применял эти техники последовательно.

Сначала использую **классы эквивалентности**, чтобы разделить значения каждого параметра на группы. Например, для суммы перевода:

- меньше минимума;
- валидный диапазон;
- больше максимума;
- пустое значение;
- некорректный формат.

Потом применяю **анализ граничных значений**, чтобы проверить точки около лимитов. Например, если перевод разрешён от 100 до 500 000 рублей, я проверю:

- `99`, `100`, `101`;
- `499999`, `500000`, `500001`.

После этого, если параметров много, использую **pairwise**. Например, у перевода есть параметры:

- тип клиента: обычный / премиум;
- валюта: RUB / USD / EUR;
- канал: web / mobile;
- способ подтверждения: SMS / push;
- тип карты: дебетовая / кредитная.

Полный перебор даст слишком много кейсов. Pairwise поможет сократить набор так, чтобы каждая пара значений встретилась хотя бы один раз.

Но в финтехе я не полагаюсь на pairwise полностью. Если есть критичная комбинация, например крупная сумма + валютный перевод + новый получатель + SMS-подтверждение, я добавлю её вручную, даже если алгоритм её не выбрал.

То есть мой подход такой: классы помогают выбрать группы данных, границы — найти ошибки на порогах, pairwise — сократить комбинации, а risk-based подход — не потерять критичные сценарии.
