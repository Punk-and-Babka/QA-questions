# 8. Объяснить пирамиду тестирования и соотнести ручные проверки с уровнями тестов
Пирамида тестирования показывает, что проверки лучше распределять по уровням, а не пытаться всё тестировать через UI.

Внизу пирамиды находятся **unit-тесты**. Их много, они быстрые и проверяют отдельные функции или бизнес-правила. Например, расчёт комиссии, округление суммы, проверку лимита.

Выше идут **интеграционные и API-тесты**. Они проверяют взаимодействие сервисов: например, сервис платежей обращается к сервису лимитов, антифроду или внешнему провайдеру.

Ещё выше — **системные и E2E-проверки**. Они проверяют полный пользовательский сценарий: пользователь создал платёж, подтвердил его, получил финальный статус, увидел операцию в истории.

Ручные проверки обычно находятся ближе к верхней части пирамиды: системное тестирование, приёмочное тестирование, исследовательское тестирование, E2E. Но manual QA не ограничивается только UI. На практике я могу руками проверить API через Postman, посмотреть запись в БД, найти логи по `operation_id`, проверить сообщение в Kafka.

Например, формулу комиссии нет смысла каждый раз проверять только через интерфейс. Лучше, чтобы она была покрыта unit/API-тестами, а вручную через UI достаточно проверить несколько ключевых пользовательских сценариев.

Главная мысль пирамиды: ручное тестирование важно, но оно не должно быть единственным способом контроля качества. Чем ниже уровень проверки, тем быстрее и дешевле находить часть дефектов.
