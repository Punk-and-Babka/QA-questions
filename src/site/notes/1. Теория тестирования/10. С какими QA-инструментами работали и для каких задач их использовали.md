# 10. С какими QA-инструментами работали и для каких задач их использовали (дописать про k8s)
Я бы разделил инструменты по задачам, потому что для QA важно не просто знать названия, а понимать, на каком уровне искать проблему.

Для задач, багов и документации использовал **Jira, Confluence, TestRail, Zephyr, Test IT**. В них ведутся user story, баг-репорты, чек-листы, тест-кейсы, тест-раны и регрессионные наборы.

Для API — **Postman** и Swagger. Через Postman удобно проверять запросы, статус-коды, тело ответа, заголовки, токены, ошибки валидации и переменные окружения.

Для веба — **Chrome DevTools**. Чаще всего использую Network, Console, Application и Elements. В Network смотрю запросы, payload, response, headers. В Application — cookies, local storage, session storage, токены. В Console — ошибки фронта. 
(Также запросы можно подменить через девтулзы)

Для базы данных — **DBeaver, DataGrip или pgAdmin**. Обычно это SELECT-запросы, JOIN, фильтрация по `user_id`, `operation_id`, статусу транзакции или времени создания операции.

Для логов и мониторинга — **Kibana, Graylog, Splunk, Grafana**. Там удобно искать ошибки по `request_id`, `trace_id`, `operation_id`, смотреть таймауты, 5xx, проблемы интеграций и поведение сервиса после деплоя. Можно отслеживать весь путь ивента.

Для брокеров сообщений — **Kafka UI, Offset Explorer, Conduktor**. С их помощью можно проверить, пришло ли сообщение в нужный топик, какая у него структура, key, offset, и обработал ли его consumer.

Если коротко, мой подход такой: UI проверяю через DevTools, backend — через Postman, данные — через БД, интеграции — через логи и Kafka, а процесс и результаты фиксирую в Jira и тестовой системе.
